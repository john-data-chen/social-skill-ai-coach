## 四、強項與風險

### 真正的差異化強項

1. **單一真相、三路消費**——大多數參賽者會把 curriculum 寫死在程式或重複貼三份；你做到一份 Skill 同時餵 in-process、MCP、drop-in CLI。這是評審「clever usage of existing toolsets」最愛看的東西。
2. **Agent 必要性論證**——你能說清楚「為何不是一個 chatbot」，而且用「Analyzer 禁止給建議、Role-Play 不討好」這種反直覺約束佐證。多數作品做不到。
3. **問題真實 + Agents for Good 契合度高**——降低一項決定歸屬感/就業/心理健康的生活技能門檻，正中賽道。

### 風險（依嚴重度排序）

1. **🔴 Demo 金鑰在審查期間可能失效。** README 已自承：MiMo token plan 審查後過期、且**未準備 DeepSeek 金鑰**。評審「親自體驗 Public Project Link」是計分項。若審查當下金鑰恰好耗盡/過期，評審打開 demo 卻不能用 → 直接傷 Core Concept & Value 與整體印象。BYOK 只在「評審剛好有 MiMo/DeepSeek 金鑰」時救得回——但這兩家是冷門 provider，**評審手上有的機率極低**。→ 這點直接連到第五節的 provider 問題。
2. **🟡 影片「執行品質」（非結構）**——逐字稿結構已驗證優秀，剩餘風險集中在製作面：腳本標註用 AI 配音 + 自動字幕。對「Agents for Good」這種情緒型 pitch，平板的機器配音會抵消強腳本；自動字幕易有錯字；手機 demo 畫面在小螢幕需確認可讀。→ 這是目前最高槓桿的執行項：用高品質人聲或精選 AI 語音（非預設音色）、人工校對字幕、確保 demo 畫面放大可讀。
3. **🟡 結構化輸出脆弱性**——reflection 靠 prompt 要 JSON 再自解析，沒有 schema enforcement，技術評審可能注意到。

## 五、回答你的問題：Demo 加入 Claude / Gemini 等 provider 會加分嗎？

### 直接結論

**會加分，但加分的「來源」不是你以為的那個。**

- ❌ 「支援更多 provider」**本身不是評分槓桿**——rubric 沒有一項在數 provider 數量。你的架構在文件上已宣稱「model-portable」，MCP 已能跑 200+ model，所以「為了展示可攜性」再加 provider，**邊際效益很低**。
- ✅ 但**加 Gemini 到 BYOK demo** 有一個真實且具體的加分理由：**讓評審能用「他手上已經有的金鑰」實際跑你的 live demo。**

### 為什麼是 Gemini，而且為什麼這點重要

1. **這是 Google 的課程（5-Day AI Agents Intensive with Google）。評審與參賽者手上幾乎一定有 Gemini API key。** 你現在 demo 只吃 MiMo / DeepSeek——評審手上有這兩者金鑰的機率近乎零。
2. 它**直接化解第四節的 🔴 風險**：就算 MiMo plan 審查期間過期，評審仍能貼自己的 Gemini key 把四階段完整跑一遍。這是在保護「Public Project Link 親身體驗」這個計分項，屬於 **防守性加分（de-risk）**，不是錦上添花。
3. 因此價值排序：**Gemini > OpenAI > Claude**（純就「評審手上有沒有金鑰」而言；課程是 Google → Gemini 命中率最高）。

### 改動的優缺點

**優點**

- 評審可用自有金鑰實跑 demo，化解金鑰過期風險。
- 兌現 README 既有 Roadmap（「Support more models: Anthropic, OpenAI, Google Gemini」），言行一致。
- 用 demo 具體證明「可攜性」，比文件宣稱更有力。

**缺點 / 成本**

- Gemini / Anthropic **不是純 OpenAI-compatible**，無法只靠現有 `@ai-sdk/openai-compatible` 一招打天下：
  - Gemini 有 OpenAI 相容端點，但 base URL / 模型名 / 邊界情況要另測；或乾脆裝 `@ai-sdk/google`。
  - Anthropic 需 `@ai-sdk/anthropic`，金鑰格式、訊息格式、structured output 行為都不同。
- 新增 provider = 新 UI 選項 + 新金鑰格式驗證 + 新的 e2e/unit 測試 + 失敗降級邏輯，**在截止前是真實的時間與回歸風險**。
- 你目前 reflection 的「自解析 JSON」策略要在新 model 上重測（Gemini 反而支援原生結構化輸出，是好事，但要驗）。

### 建議與信心水準（原始版）

- **若距截止還有充裕時間：做「Gemini-only」的 BYOK 最小支援**（一個 provider、一個下拉選項、跑通四階段 + reflection JSON 解析），不要一次塞三家。**建議執行，信心 70%。** 理由：它的價值是「防守 demo 可用性」，不是「多即是好」。
- **若時間緊：不要碰 provider，把同樣時間投到影片品質與 writeup 收尾（合計 20 分且第一印象權重高）。** 對「provider 不是高 CP 值投資」這個判斷，**信心 80%**。
- **Claude / OpenAI 全加：邊際效益最低，截止前不建議。信心 75%。**

> 一句話：**不是「加 provider 加分」，而是「加 Gemini 讓評審能親手把 demo 跑起來」才加分。** 把它當作「demo 可用性保險」來做，而不是「功能展示」。

### ⚠️ 修正判斷（2026-06-24）：Gemini BYOK **不值得做**

上述建議在以下事實反駁後**不再成立**：

1. **Gemini 訂閱 ≠ API 存取**：即使已訂閱 Google Gemini 年費（Google One AI Premium / Gemini Advanced），仍**無法以 API 形式使用**，必須另外到 Google AI Studio 或 GCP 開 billing。這直接瓦解「評審手上一定有 Gemini API key」的核心假設。
2. **成本不對等**：Gemini API 最低購買成本約 NT$400（~US$13），對比 MiMo US$2 / DeepSeek US$6，高出 2–6 倍。而本專案已投入 US$20（Claude）+ US$24（AI 影片製作）= US$44+，再花 US$13 驗證一個邊際效益 0–2 分的功能，CP 值極低。
3. **「評審有 Gemini API key」機率被高估**：課程使用 Google AI Studio 免費額度，課程結束後多數人未綁 billing，實際機率可能僅 30–40%，非原先隱含的 80%+。
4. **實作風險**：新 provider = 新 bug surface，截止前引入回歸風險反而危險。

**修正結論**：

- 做了能加 **0–2 分**（Rubric 無「provider 數量」項），僅在評審碰巧有 key + MiMo 碰巧掛掉時才有防守價值。
- **能衝前三嗎？不能。** 前三決定因素是影片品質、pitch 情緒感染力、評審主觀印象——全跟 Gemini BYOK 無關。
- **更好的零成本防守策略**：確認 MiMo token plan 覆蓋審查窗口 + 備妥 DeepSeek key + README 標示清楚 BYOK 步驟。

| 不做 Gemini、時間投影片品質更划算 | 80%    | **90%**  |
